Modulstruktur — drei klare Schichten

Jedes Modul hat einen Zweck, ein Interface, eine Testdatei. Kein Modul kennt mehr als seine direkten Nachbarn.

Verzeichnislayout
ki_geodaten/
├── app/                       # Schicht 1: UI (FastAPI + Leaflet)
│   ├── main.py                # FastAPI-App, Route-Registrierung
│   ├── routes/
│   │   ├── jobs.py            # POST /jobs, GET /jobs, /jobs/{id}/status
│   │   ├── validation.py      # POST /jobs/{id}/polygons/{pid}/accept|reject
│   │   └── export.py          # GET /jobs/{id}/export.gpkg
│   ├── templates/index.html   # Leaflet-Karte, Prompt-Eingabe, BBox-Zeichner
│   └── static/                # app.js, style.css
│
├── pipeline/                  # Schicht 2: Reine Bildverarbeitung
│   ├── wcs_client.py          # DOP20-Download von Bayern LDBV WCS
│   ├── tiler.py               # rasterio-basiertes Tiling + Affintransform
│   ├── segmenter.py           # SAM 3.1-Wrapper: (Tile, Prompt) → Masken+Boxes+Scores
│   ├── merger.py              # Polygon-NMS via Shapely über Kachel-Grenzen
│   ├── georeferencer.py       # Pixel-Polygone → EPSG:25832 (GeoPandas)
│   └── exporter.py            # GeoDataFrame → GeoPackage
│
├── jobs/                      # Schicht 3: Orchestrierung & Persistenz
│   ├── orchestrator.py        # Führt die Pipeline-Schritte sequenziell aus
│   ├── store.py               # SQLite: Jobs, Polygone, Status, Validierungs-Flags
│   └── models.py              # Pydantic: Job, Polygon, BBox, JobStatus
│
├── config.py                  # Pfade, WCS-URL, Modell-Checkpoint
├── data/
│   ├── jobs.db                # SQLite
│   ├── dop/{job_id}.tif       # heruntergeladene Orthophotos
│   └── results/{job_id}.gpkg  # Export
│
└── tests/
    ├── test_tiler.py          # Fixture-GeoTIFF, bekannte Affintransformation
    ├── test_merger.py         # synthetische überlappende Polygone
    ├── test_georeferencer.py  # Pixel → UTM Transformation
    └── test_wcs_client.py     # Mock-WCS-Antworten

Warum diese Aufteilung?

Die drei Schichten sprechen in eine Richtung

  • app/ ruft jobs/ — aber nicht pipeline/ direkt
  • jobs/ orchestriert pipeline/-Module
  • pipeline/ kennt weder UI noch Jobs — reine Funktionen
  • → Pipeline-Module sind per CLI/Notebook unabhängig testbar

Jedes pipeline/-Modul

  • hat klar definierte Ein-/Ausgaben (typisiert via Pydantic)
  • ist ohne Seiteneffekte implementiert (außer I/O-Modulen)
  • hat eine 1:1-Testdatei
  • darf bei Bedarf ein eigenes Fachvokabular nutzen, aber exportiert nur einfache Dataclasses

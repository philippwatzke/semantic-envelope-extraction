Überarbeitete Architektur — zwei Prozesse, eine GPU

Die harte Trennung von Webserver und GPU-Worker löst alle vier Probleme zugleich.

Wie jedes deiner vier Probleme gelöst wird

① VRAM-Kollaps → Physikalisch unmöglich

Nur ein Worker-Prozess läuft. Concurrency=1 ist keine Vereinbarung, sondern eine Eigenschaft des OS. Der Worker lädt SAM 3.1 einmalig beim Start und behält es im VRAM — das spart die Lade-Zeit (~8 s) pro Job und garantiert, dass kein zweites Modell entsteht. Die Job-Queue ist einfach eine SELECT ... WHERE status='PENDING' ORDER BY created_at LIMIT 1-Abfrage.

② Blocking → Webserver sieht die Pipeline nie

POST /jobs macht exakt drei Dinge: Request validieren, Zeile in SQLite anlegen (Status PENDING), Job-ID zurückgeben. Antwortzeit: <50 ms. Der Webserver weiß nicht, ob überhaupt ein Worker läuft. Der Client pollt GET /jobs/{id}/status — keine Long-Polls, keine WebSockets, keine Threads im Webserver.

③ WCS-Pagination → Zwei-Ebenen-Tiling
Ebene A (Download-Tiling): wcs_client.py kennt MAX_WCS_PIXELS = 4000. Bei größeren BBoxes: Raster aus Sub-Requests, jeweils ≤4000×4000 px, dann lokal via rasterio.merge zu einem GeoTIFF zusammengesetzt. Affintransformation bleibt konsistent.

Ebene B (Inference-Tiling): tiler.py operiert auf dem gemergten GeoTIFF und produziert 1024×1024-Kacheln mit 256 px Overlap für SAM 3.1. Unabhängig von WCS-Limits.
④ OOM/Crash-Handling → Worker-Resilienz
Drei Absicherungen, in dieser Reihenfolge:
  • Worker-Startup-Hook: markiert alle Jobs mit Status PROCESSING als FAILED (worker restarted) — befreit die Queue nach Crash.
  • Globales try/except um orchestrator.run(job): jeder Fehler (inkl. torch.cuda.OutOfMemoryError) landet im Status FAILED mit Stacktrace in jobs.error_message, sichtbar in der UI.
  • Per-Tile-try/except in segmenter.py: wenn eine Kachel OOM wirft, wird sie übersprungen (torch.cuda.empty_cache()), der Job liefert Teilergebnisse statt ganz zu scheitern.
